# AGENTS.md

> Equivalent to `CLAUDE.md`. If you change workflow rules, update both.

## Agent Contract

This file is a compact routing contract. Operational detail lives in `scripts/workflow.py`, the Agent Skills under `.claude/skills/` and `.agents/skills/`, and the active slice folder.

Core rule: **Backlog routes. Slice folder explains. Result summarizes. Docs are versioned durable truth.**

## Driving This Workspace

Everything runs through one manager: `python3 scripts/workflow.py <command>`. The same operations are also packaged as Agent Skills so they work natively in either tool:

- **Claude Code:** slash commands like `/create-phase`, `/do-next-slice`, `/do-whole-phase`, `/review-phase` (from `.claude/skills/`), plus the `slice-executor-low` / `slice-executor-mid` / `slice-executor-high` subagent tiers that implement delegated slices and run the phase review. `.claude/settings.json` pre-approves the workflow script so it runs without prompts.
- **Codex:** the same skills under `.agents/skills/` via `$skill` or `/skills` ‚Äî **except `do-whole-phase`, which is Claude Code only** (Codex has no plan mode, so it cannot do genuine per-slice planning across that loop) ‚Äî plus the same `slice-executor-low/-mid/-high` subagent tiers under `.codex/agents/` that implement delegated slices and run the phase review (on their configured models, `.codex/agents/*.toml`). Codex reads this file as `AGENTS.md`.
- **Any agent / CI:** call `python3 scripts/workflow.py ...` directly. This always works, even where skills are unavailable.

**Orchestrator and executor.** `do-next-slice` and `do-whole-phase` run as an orchestrator/worker split. The **orchestrator** (main thread) plans each slice at the operator's gate (plan mode by default ‚Äî research, clarify, confirm), writes its **native plan** to `plan.md` (free-form, no template), transitions workflow state, commits, and talks to the operator. Every slice ‚Äî decomposition, implementation, `fix`, and the phase review ‚Äî is delegated to one of three **`slice-executor`** tiers (Claude Code: `.claude/agents/slice-executor-{low,mid,high}.md`; Codex: `.codex/agents/slice-executor-{low,mid,high}.toml`), picked by the slice's `kind` + `risk`: decomposition, review, and high/unknown-risk slices ‚Üí `slice-executor-high` (opus @ `xhigh` by default), `risk` exactly `medium` ‚Üí `slice-executor-mid` (sonnet @ `xhigh`), exactly `low` ‚Üí `slice-executor-low` (haiku ‚Äî a literal plan-follower: no judgment calls, no improvisation; route to it only fully mechanical plans), while the orchestrator stays `max`; the orchestrator may bump a slice one tier up (never down), and tier models/efforts are configurable via a repo-root `executors.toml` applied with `python3 scripts/workflow.py sync-agents`. The executor does the slice's job against `plan.md`, writes `result.md`, appends cross-slice notes to `phase.md`, and returns a structured verdict ‚Äî it never commits and never transitions slice/phase status (it may run `new-slice` only while executing a decomposition slice, and `doc-new-version` only while executing the review slice). Delegating the execution ‚Äî including decomposition's slice-creation and the phase review ‚Äî is what keeps the orchestrator's context lean. In Claude Code the orchestrator always dispatches the executor **as a background task** (the Agent tool default ‚Äî never `run_in_background: false`) and waits for its completion notification, doing nothing else in the meantime; it never runs the executor synchronously inline in the main session. By default the orchestrator advances on its own ‚Äî after each slice it automatically plans the next; the only pause is the operator's approval of each **readied plan** before the executor runs (**plan ‚Üí operator approves the readied plan ‚Üí executor, repeated** ‚Äî never a pause before planning begins). The operator can invoke the skill with `auto` to skip that plan-approval and run plan ‚Üí executor straight through (the safety halts ‚Äî `pending`, `needs_operator`, `blocked`, a failed `slice-executor-high` return ‚Äî still STOP the loop; an `escalate` or a failed low/mid return re-dispatches up the ladder instead of stopping). Both skills also accept **`plan only`** (e.g. `/do-whole-phase plan only`): the orchestrator readies operator-approved plans without dispatching executors ‚Äî each approved plan is written to the slice's `plan.md` and the slice is set `ready` (shown `[r]`); `do-whole-phase plan only` first ships the `DECOMP` slice as usual when it isn't done (the middle slices must exist to be planned), then readies every middle slice and stops before `REVIEW`, which is never pre-planned; `plan only` never combines with `auto`. A `ready` slice is pre-approved: when a later run reaches it, the orchestrator skips plan mode and dispatches the executor directly from the existing `plan.md`, re-planning only if the workspace state has drifted. The orchestrator **trusts a `done` verdict** ‚Äî it does not re-run the slice's validation ‚Äî and runs only `validate` (state integrity) before it finishes and commits; behavioral validation is consolidated into the phase review, which validates all slices at once and consolidates the phase's durable-doc versions. A `needs_operator`, `blocked`, failed, or empty return still means the slice is not done. A low/mid-tier executor that finds a slice beyond its depth returns **`escalate`** with its findings; the orchestrator appends them to the slice's `plan.md` as an `## Escalation` section and re-dispatches the next tier up (`low ‚Üí mid ‚Üí high`; a failed/empty low/mid return is treated the same; at most 2 escalations per slice; the top tier never escalates), re-presenting the revised plan for operator approval unless running `auto`.

Workflow command-skills are explicit-invocation only; agents should not fire them autonomously.

**Capture intent first.** When an operator request arrives, before acting: **refine** it into clear language, **clarify** anything ambiguous by asking the operator, and **confirm** your understanding. Only after the operator confirms do you act. This applies wherever operator intent first enters a unit of work ‚Äî always at the phase level, and at the slice level when an operator note is ambiguous.

**Making a phase ‚âÝ executing it.** When the operator asks you to make, create, suggest, or plan a phase, use the `/create-phase` skill: capture intent, then run `new-phase` ‚Äî which creates only `P<N>.DECOMP` and `P<N>.REVIEW` and scaffolds the phase-root `intent.md` (linked from `phase.md`) ‚Äî then fill that `intent.md` with the confirmed intent, and then STOP and report. Do **not** decompose the phase into middle slices, do **not** write slice plans, and do **not** implement any code. Decomposition is the `DECOMP` slice's own job and happens later, when the operator executes the phase (`/do-next-slice`, `/do-whole-phase`) or explicitly tells you to. Creating several phases at once is fine; decomposing or executing any of them is a separate, explicit step.

## Canonical State

- Current pointer: `works/state.json`
- Generated dashboards/index: `works/backlog.md`, `works/deferred.md`, `works/index.json`
- Phase state: `works/phases/active/<phase_id>/phase.json`
- Phase notebook: `works/phases/active/<phase_id>/phase.md` ‚Äî objective plus the accumulating decomposition, findings, cross-slice notes, and a running "Doc impact" list (durable-truth changes the review slice consolidates into doc versions); the shared context across a phase's slices
- Phase intent: `works/phases/active/<phase_id>/intent.md` ‚Äî scaffolded by `new-phase` and linked from `phase.md`; holds the operator's verbatim original request plus the confirmed refined intent (and resolved clarifications), filled by the `/create-phase` skill. The source of truth for what the operator asked when intent is unclear.
- Slice state: `works/phases/active/<phase_id>/slices/<slice_id>/slice.json`
- Slice context: `plan.md` (the orchestrator's free-form native plan for the slice, written at the slice's turn ‚Äî no fixed template) and `result.md` (the executor's free-form result, written from scratch at slice end ‚Äî no template either), beside `slice.json`
- Deferred state: `works/deferred/open/<DID>/deferred.json`
- Doc index: `docs/index.json`; latest docs: `docs/current/*.md` generated from `docs/versions/<doc>/vNNNN_*.md`

## Hard Rules

- Keep `works/backlog.md` and `works/deferred.md` lean: IDs, names, statuses, pointers, paths only. Detail goes in the folders.
- Keep test files small by default: tests are welcome, but keep each test file or suite terse ‚Äî minimal high-value cases, no fixture or scaffolding sprawl. Prefer lightweight verification (run the code, `validate`, a small smoke check); grow a suite only when the operator asks or the risk clearly warrants it.
- Never patch old files under `docs/versions/`; create a new version with `doc-new-version`.
- Treat `docs/current/*.md` as generated snapshots; never hand-edit them.
- Durable docs are versioned **once per phase, at the review slice**: implementation, `fix`, and decomposition slices never run `doc-new-version` ‚Äî when they change durable truth they append a one-line "Doc impact" note to `phase.md`, and the review slice consolidates those into new versions on a passing review.
- New phases start with only `P<N>.DECOMP` and `P<N>.REVIEW`, plus a scaffolded phase-root `intent.md` (linked from `phase.md`). The `DECOMP` slice ‚Äî planned by the orchestrator, executed by `slice-executor-high` ‚Äî creates the middle slices **only** ‚Äî bare folders ‚Äî and records the slice breakdown, findings, and notes in `phase.md`; it does **not** pre-fill the new slices' `plan.md`. It sets each middle slice's `--risk` deliberately: `risk` selects the executor tier, so it is the phase's main cost lever (`low` only for fully mechanical work).
- "Make/create/suggest a phase" = use the `/create-phase` skill (captures intent, then runs `new-phase`, which creates `DECOMP` + `REVIEW` and scaffolds `intent.md`), then stop ‚Äî do not decompose, write slice plans, or implement until the operator executes the phase or says to. See *Driving This Workspace*.
- Capture operator intent at phase creation: **refine ‚Üí clarify (ask the operator) ‚Üí confirm**, then fill the `intent.md` that `new-phase` scaffolds in the phase folder (verbatim original + confirmed refined intent + any resolved clarifications); the engine links it near the top of `phase.md`. Do **not** run `new-phase` until the operator confirms your understanding. The verbatim original is immutable; only the confirmed wording is refined.
- When unsure of the operator's intent, consult the phase's `intent.md` (linked from `phase.md`) ‚Äî the confirmed source of truth for what was asked. For slice-specific intent, read that slice's `plan.md`.
- Each slice owns exactly two context files: `plan.md` (the orchestrator's free-form native plan for the slice, written at the slice's turn ‚Äî no fixed template; it incorporates any operator note passed with `do-next-slice`/`do-whole-phase`, while the operator's verbatim intent lives in the phase's `intent.md`, not duplicated per slice) and `result.md` (the executor's free-form result, written from scratch when done ‚Äî no template either). Neither file is scaffolded: a fresh slice folder holds only `slice.json`. A slice never pre-fills another slice's `plan.md`. There are no per-slice brief or review files.
- `phase.md` is the phase notebook: the `DECOMP` slice seeds it (breakdown, findings, notes), and every slice reads it for accumulated context at start and appends durable cross-slice notes back to it when it finishes ‚Äî so later slices build on what earlier ones learned.
- Every slice ‚Äî decomposition, implementation, `fix`, and the phase review ‚Äî is executed by a dispatched `slice-executor` tier (`slice-executor-low` for implementation/`fix` with `risk` exactly `low` ‚Äî a literal plan-follower that escalates on any surprise; `slice-executor-mid` for exactly `medium`; `slice-executor-high` for decomposition, review, and everything else; defaults haiku / sonnet@`xhigh` / opus@`xhigh`, configurable via the repo-root `executors.toml` + `sync-agents`) ‚Äî not by the orchestrator directly. The orchestrator owns plan-mode planning, every status transition, and all commits; the executor does the slice's job, writes `result.md`, and appends the slice's `phase.md` notes, but never commits and never transitions slice/phase status (it may run `new-slice` only during a decomposition slice, and `doc-new-version` only during the review slice). `plan.md` is the orchestrator's free-form native plan, written at the slice's turn (just-in-time, no template); `result.md` is likewise free-form, written from scratch by the executor. The orchestrator trusts a `done` verdict and re-runs only `validate` (state integrity), not the slice's tests; behavioral validation and durable-doc consolidation are done by the phase review ‚Äî which the executor runs and the orchestrator records via `review-phase`. On an `escalate` verdict (or a failed/empty low/mid return) the orchestrator appends the findings to the slice's `plan.md` as an `## Escalation` section and re-dispatches one tier up ‚Äî never down, never past `slice-executor-high`, at most twice per slice.
- Slice selection is by `order`; `--order` accepts fractional values (e.g. `--order 4.5`) so a slice (or phase) can be inserted between two existing neighbors without renumbering. `depends_on` is advisory and only checked for existence by `validate`.
- Operator co-work (`pending`, shown `[~]`): when a slice or phase needs the operator ‚Äî to validate something, or to run an action only the operator can perform ‚Äî set it `pending` (`set-slice-status <id> pending` or `set-phase-status <P> pending`), report exactly what you need, and STOP. A `pending` item halts selection: `next` prints `WAITING ON OPERATOR`, and neither `do-next-slice` nor `do-whole-phase` may start, finish, or advance past it. Work resumes only after the operator approves ‚Äî they (or you, on their explicit say-so) clear it with `set-slice-status <id> in_progress` (or `set-phase-status <P> in_progress`). `pending` means "waiting on the operator" and is distinct from `blocked` (an impediment or unmet dependency you cannot resolve yourself).
- Plan-ready slices (`ready`, shown `[r]`): the `plan only` mode of `do-next-slice`/`do-whole-phase` writes the operator-approved plan to a slice's `plan.md` and sets it `ready` (`set-slice-status <id> ready`) instead of executing; `do-whole-phase plan only` ships `DECOMP` first when needed and stops before `REVIEW`, which is never pre-planned; `plan only` never combines with `auto`. `ready` means pre-approved: execution runs dispatch the executor straight from the existing `plan.md` without re-entering plan mode (re-planning only if the workspace state has drifted), and `start-slice` moves it to `in_progress`. `validate` errors on a `ready` slice that has no `plan.md`.
- Deferred jobs never affect next-slice selection until promoted.
- Record the phase review with `review-phase` ‚Äî the verdict drives both the phase and its `REVIEW` slice: `pass` ‚Üí phase `done` + `REVIEW` slice `done`; `changes_requested` ‚Üí phase `in_progress` + `REVIEW` slice `changes_requested` (reopened for re-review); `blocked` ‚Üí both `blocked`. A passing review marks a phase `done` but does **not** archive it ‚Äî the phase stays in `active/`. Archiving is a separate, manual step: `archive-all` once every active phase is done (the last review slice complete), `rotate-backlog` to archive just the done phases while others continue, or `archive-phase <P>` for a single review-passed phase. Archive whole phases only, never individual slices.
- Upstream bootstrap repo only (where `installer/` exists ‚Äî adopting workspaces have none, so this rule is inert there): after editing any embedded machinery file (`scripts/workflow.py`, `.claude/*`, `.agents/*`, `.codex/*`, `works/templates/*`, or this contract), run `python3 installer/build.py` and commit the rebuilt `bootstrap_agentic_workspace.sh` in the same commit ‚Äî `python3 installer/build.py --check` must pass before committing. The tracked `.githooks/pre-commit` hook enforces the check; register it once per clone with `git config core.hooksPath .githooks`.

## IDs and Status

- Phase IDs: `P1`, `P2`, ... with status `planned | in_progress | in_review | pending | blocked | done`
- Slice IDs: `P1.DECOMP`, `P1.S1`, `P1.F1`, `P1.REVIEW`, ... with status `todo | ready | in_progress | in_review | changes_requested | pending | blocked | done`
- Deferred IDs: `D1`, `D2`, ... with status `deferred | ready | promoted | done | dropped`
- Doc versions: `v0001_bootstrap.md`, `v0002_<slug>.md`, ...
- Phase review verdicts: `pass | changes_requested | blocked`

## Workflow Commands

Use `python3 scripts/workflow.py <command>`:

- `next` ‚Äî show current/next active slice
- `new-phase --phase P2 --name "..." --objective "..."` ‚Äî also creates `DECOMP` + `REVIEW` and scaffolds the phase-root `intent.md`; the `/create-phase` skill captures intent and fills it
- `new-slice --phase P1 --slice P1.S1 --name "..."` (`--kind`, `--risk`, `--order`, `--depends-on`)
- `start-slice P1.S1` / `finish-slice P1.S1` / `set-slice-status P1.S1 <status>`
- `set-phase-status P1 <status>`
- `set-slice-status P1.S1 pending` / `set-phase-status P1 pending` ‚Äî hand off for operator co-work (validation or operator-run action); clear with `... in_progress` after approval
- `review-phase P1 --verdict pass|changes_requested|blocked [--reviewer NAME] [--note "..."]` (also transitions the phase's `REVIEW` slice per the verdict)
- `doc-new-version --doc frontend --summary "..." --source P1.REVIEW` / `docs` / `rebuild-docs` ‚Äî durable-doc versioning; run at the phase review to consolidate, not per slice
- `deferred` / `defer-job --title "..." --reason "..." --trigger "..." --source P1.S1`
- `promote-deferred D1 --phase P1 --slice P1.S2 --name "..."` / `drop-deferred D1 --reason "..."`
- `archive-all` (batch-archive every active phase once all are done) / `rotate-backlog` (archive just the done phases, leave the rest) / `archive-phase P1` (archive a single review-passed phase)
- `rebuild` / `validate`
- `sync-agents [--check]` ‚Äî apply the repo-root `executors.toml` executor-tier config (models/efforts) to the six `slice-executor-*` agent files; run after editing `executors.toml` and after a workspace update (`--check` reports drift without writing)

## Commit Convention

Use `type(scope): summary`, imperative voice, no trailing period. Types: `feat`, `fix`, `docs`, `chore`, `refactor`, `test`, `ci`, `build`, `perf`, `revert`.

By default, commit after each completed slice ‚Äî at the end of `do-next-slice`, and at every clean slice boundary inside `do-whole-phase` (Claude Code only). Outside the slice workflow, commit only when asked. **Attribute each commit to the model that actually did the work.** Claude Code adds its own `Co-Authored-By: Claude ‚Ä¶` trailer automatically (naming whichever model ran the session); in another tool, add a `Co-Authored-By` trailer naming the model that ran ‚Äî e.g. in Codex, `Co-Authored-By: GPT-5.5 <noreply@openai.com>`. Never attribute a commit to a model that didn't do the work (e.g. don't carry over another tool's trailer). Do not create branches unless the operator asks ‚Äî work on the current branch, including `main`. Never push without being asked.

Auto-explain at the phase review: how a bootstrap workspace learned to document its own phases

A change-mode explainer for phase P8 of the bootstrap_agentic_workspace.sh repo. It walks what P8 changed, why, and how the design holds together — including the fact that this very document was produced by the machinery P8 added.

Background

This repository builds a self-installing "agentic workspace": a set of conventions and a small Python engine (scripts/workflow.py) that let AI coding agents drive real software work in disciplined units. If you have never seen it, here is the one-paragraph mental model you need before the change makes sense — skip ahead to the review step if you already know it.

The vocabulary: phases, slices, and the orchestrator/executor split

Work is organized into phases (a unit of intent, like "add feature X"), and each phase is cut into slices (small, individually shippable steps). A phase always begins with a DECOMP slice (which plans the middle slices) and ends with a REVIEW slice. Two kinds of agent cooperate:

  • The orchestrator — the main thread. It plans each slice, transitions workflow state, talks to the operator, and owns every git commit.
  • The executor — a dispatched sub-agent (slice-executor-low / -mid / -high, chosen by the slice's risk). It does the slice's actual work in an isolated context, writes a result.md, and returns a verdict. Crucially, an executor never commits and never transitions state — that boundary is what keeps the main thread's context lean and the workflow auditable.
The phase review, before P8. The REVIEW slice is run by the top-tier slice-executor-high. Its job was two things: (1) validate all of the phase's slices together (re-running their checks), and (2) on a passing review, consolidate documentation — turning the running "Doc impact" notes that earlier slices left in phase.md into new versioned doc entries. In shorthand: review = validate + consolidate docs.

Where "explain" comes from

Separately, the project long carried an /explain skill: given a topic or a code change, it researches the real code and writes a single self-contained interactive HTML "explainer" — Background, Intuition, Code, a cited best-practices section, and a quiz — filed into a personal knowledge base (KB). Originally this skill was embedded in the bootstrap and shipped to every adopting workspace. But it was coupled to one person's machine (a KB at a fixed path, an API on localhost), so a prior phase, P7, retired the embedded copy and moved explain into a portable external Claude Code plugin (in a separate "knowledge" repo). After P7, the bootstrap no longer ships explain at all.

That set up an obvious gap. The operator's original request had two halves: make explain portable (done by P7 + the knowledge repo's own phases), and "for all review slices, want to use the skill automatically… basically phase review + docs + and the explain." P8 is that second half — wiring the now-external explain skill back into the review, so a passing review also produces a phase explainer.

Intuition

The essence of P8 is a single sentence: a passing phase review now also produces a phase explainer, so the review's job grows from validate + consolidate docs to validate + consolidate docs + explain. Everything else is about doing that safely — because the machinery ships to every adopting workspace, most of which have no knowledge plugin and no KB at all.

REVIEW slice runs → validate all slices → verdict?

On pass only:

consolidate docs → auto-explain → skip / saved (best-effort) → record verdict

On changes_requested / blocked: no docs, no explainer — fixes land first.

Five design decisions make the sentence work. Think of them as answers to five sharp questions:

QuestionDecision
Who runs it? The slice-executor-high review executor — by locating the installed explain skill and following it, never copying its ~500 lines into the machinery (that would drift from the plugin). The full procedure lives in one place, the review checklist; the agent files carry only a short pointer.
What if it isn't installed? An ordered search — project, then user, then plugin installs — with graceful skip. Not found (or the KB probe says "unconfigured") → skip with a note.
When does it fire? Pass-gated: only on a passing review, exactly like doc consolidation. And verdict-neutral — the explain outcome never changes the review verdict, even on failure.
Can it research the web? Yes, but only where safe: WebSearch and WebFetch were added to the Claude high executor only, so the cited "best practices" section can run. Without them (e.g. Codex) it degrades to skipped-offline.
Doesn't it need to commit? Only in the rare offline fallback — and only in the separate KB repo. A narrow carve-out to the "executor never commits" rule allows exactly that, never in this workspace, never a push.

Why "best-effort" is the whole game

The key intuition is that the explainer is a garnish, not a gate. Doc consolidation is load-bearing (the docs must stay true), but the explainer is a teaching artifact — nice to have, never required. So its outcome is reported as a single line (explain: saved … / skipped … / failed …) and is forbidden from touching the verdict. A review of a real, correct phase must never turn red because a KB server happened to be down. Hold on to that idea — it is exactly the "continue-on-error" pattern the Best-practices section compares against.

Code

P8 was deliberately a single atomic slice (P8.S1), because the project has a non-negotiable rule: a machinery edit and the rebuilt single-file installer artifact must land in the same commit (a pre-commit hook enforces it). Splitting would leave a "lying" intermediate state. So one commit, feat(review): auto-explain the phase at a passing review, touches the checklist, the agent files, the contract, the version, and the rebuilt artifact together. Let us walk the real diff, grouped logically.

1. The heart: the review checklist gains one paragraph

The whole procedure lives in .claude/skills/review-phase/SKILL.md (and its byte- identical .agents/ mirror), inserted right after the existing doc-consolidation paragraph — its natural home, since that paragraph already governs the pass-gated consolidation step. The new paragraph, verbatim from the diff, defines detection order, the read-and-follow rule, graceful skip, the commit exception, and the outcome vocabulary:

On a **passing** review, after consolidating docs, also **auto-explain
the phase** — produce a phase explainer via the knowledge plugin's
installed explain skill. This is best-effort: like doc versions it fires
**only on a passing review** ... and its outcome **NEVER** changes the
`review_verdict`. **Locate the installed explain skill** (first hit wins):
project `.claude/skills/explain/SKILL.md` -> user
`~/.claude/skills/explain/SKILL.md` -> plugin installs (a bounded search
... covering `cache/` and `marketplaces/`). None found -> skip and report
`explain: skipped (skill not installed)`. Found -> **read that `SKILL.md`
and follow its instructions as written** (never duplicate its procedure
here ...), in **change mode with this phase as the change-ref**.
Why reference, never duplicate? The explain skill lives in a different repo and evolves on its own. If the checklist pasted a copy of its steps, the two would silently drift apart. "Locate and follow the installed skill" makes the plugin the single source of truth — the review just knows how to find it and when to run it.

2. The high executor: three small edits

.claude/agents/slice-executor-high.md changed in three ways. First, its tools: line gained two entries:

tools: Read, Edit, Write, Glob, Grep, Bash, WebSearch, WebFetch

These are read-only research tools, so they cannot mutate state or commit — a good fit for a review executor's read-heavy job, and they let the explain skill's cited best-practices section actually run. A subtle safety check made this cheap: the sync-agents command, which keeps agent files in step with executors.toml, only ever rewrites the model:/effort: lines — so a hand-added tools: line survives a sync and does not trip the drift checker. Only the high tier got them (reviews always run there); mid/low and the Codex executors were left untouched.

Second, its review bullet gained a short pointer back to the checklist (the "single source of truth" discipline). Third — and most interesting — its "never commit" rule gained a scoped exception:

- commit or push (no `git commit`, `git add`, `git push`) — the
  orchestrator owns commits; the one scoped exception: a **review** slice's
  auto-explain offline fallback may run the explain skill's own
  `git -C <KB_ROOT> add/commit` in the **separate knowledge-base repo**
  (never in this workspace's repo, and never any `git push`);

The three KB states, and which one needs the carve-out:

unconfigured → skill STOPs at its probe → skip. No git.
API reachable → API returns 201 and makes its own server-side commit → executor runs no git at all.
API down, local KB exists → skill writes + commits with git -C <KB_ROOT>. Only this path needs the carve-out.

The reasoning behind the carve-out is precise: the "orchestrator owns commits" rule exists to keep this workspace's slice boundary clean. A commit in a different git root (the KB) never crosses that boundary, and the explain skill's offline path is designed to always fall through to a durable, committed save. Narrow enough to not weaken the bright-line rule; broad enough to let the designed fallback work.

3. The Codex mirror: asymmetry accepted

.codex/agents/slice-executor-high.toml received the same body edits (pointer, carve-out, an explain: verdict field) mirrored into its developer_instructions — but no tools key and no web tools. Codex governs tools via its sandbox_mode, not a per-agent list, so its review simply degrades the research section to skipped-offline, and under its workspace-write sandbox the offline file-write fallback is an automatic skip too. Both are exactly the graceful degradation the skill is designed for.

4. Contract, driving skills, version, and the rebuild

  • The contract files CLAUDE.md and AGENTS.md (kept body-identical from line 4, each with its own title/pointer header) had their review sentence extended to "…consolidates the phase's durable-doc versions, and — on a pass — produces the phase explainer via the knowledge plugin's explain skill."
  • The driving-skill review paragraphs (do-whole-phase, do-next-slice + its mirror) were updated the same way.
  • installer/main.py bumped WORKSPACE_VERSION 15 → 16, with a new top ## v16 CHANGELOG.md entry (historical entries never patched).
  • python3 installer/build.py regenerated the single-file bootstrap_agentic_workspace.sh, staged in the same commit; build.py --check confirms the artifact matches source.
Dogfooding. The REVIEW slice for P8 ran on the new machinery it had just landed. It validated all slices, consolidated two doc versions (operations v0016, decisions v0022), and then executed the auto-explain step for the first time in anger — producing the very page you are reading.

Best practices & next steps

P8 is, at heart, a tiny docs-as-code + CI automation design, expressed in an agent workflow rather than a YAML pipeline. It is worth measuring against how the wider industry handles the same problems.

Where P8 aligns with prevailing practice

  • Documentation is versioned and reviewed like code. The whole workspace treats durable docs as version-controlled artifacts consolidated at a review gate — exactly the docs-as-code creed that "everything lives in your Git repository… apply the same review standards to documentation as code changes" (Fern, "Docs-as-code" — buildwithfern.com). P8 extends this to a generated artifact produced at merge/review time.
  • Generate documentation at the merge/review boundary. A common industry pattern is that when a change lands, CI regenerates docs and drafts changelog/release material from the change itself (Doc Holiday, "Update Documentation Automatically When a PR Merges" — doc.holiday). P8's pass-gated explainer, built in change mode from the phase's real diff, sits squarely in this family.
  • The optional step is best-effort, not a gate. P8's verdict-neutral rule is textbook continue-on-error: an attribute that lets "task execution proceed even when a specific task encounters an error… useful when a task failure is undesirable but not fatal to the release" (Praveen Kumar, "How to use ContinueOnError" — praveenkumarsreeram.com). The explainer is precisely a non-critical step that must never block the review.
  • Graceful degradation as a first-class design. Reducing a non-essential feature to preserve the core service during partial failure is the definition of graceful degradation (SRE School, "What is Graceful degradation?" — sreschool.com). P8's ordered skill detection, KB probe, and offline fallback are a small ladder of exactly such reductions.

Where P8 deliberately diverges (and why that is legitimate here)

  • Automation as a finished publish, not a supervised first draft. Industry guidance stresses that generated docs should be "a first draft that gets refined, not a final publish that runs unsupervised," with a human editing before publication (Doc Holiday — doc.holiday). P8 files the explainer straight into the KB with no human gate. That is acceptable because the artifact is a teaching explainer in a personal knowledge base, not customer-facing product documentation — the load-bearing docs still go through the reviewed consolidation step; the explainer is downstream of it.
  • A carve-out in a bright-line safety rule. Most pipelines never let a non-privileged step commit at all. P8 opens a keyhole exception (KB repo only, offline only, no push). The divergence is justified by scoping it to a different git root, so the workspace's own commit-ownership invariant is untouched.

Two to four concrete next steps

  1. Add a lightweight integration smoke for the detection ladder. A tiny test that stubs the three skill locations and asserts first-hit-wins + the graceful-skip line would lock in the behavior the way retrofit_smoke.sh locks in installer invariants — matching the docs-as-code advice to "run automated tests" as part of the pipeline (Fern — buildwithfern.com).
  2. Record the explain outcome in phase state, not just prose. Today the explain: line lives in result.md. Capturing it as a structured field (saved-url / skip-reason) would let the operator later see which phases have explainers and re-run the misses — the observability half of continue-on-error, which recommends surfacing the failure for "manual review" rather than only swallowing it.
  3. Offer a batch back-fill. Phases reviewed before v16 have no explainer; a one-shot command to auto-explain a range of past phases would let an adopter who installs the KB later catch up, without re-reviewing.

Quiz

1. After P8, what does a passing phase review do?

The review grew from validate + consolidate docs to validate + consolidate docs + explain. Committing/archiving remain the orchestrator's job, not the review executor's.

2. The auto-explain step fails because the KB server is down. What happens to the review_verdict?

Auto-explain is best-effort and verdict-neutral — like a continue-on-error CI step, its failure is reported (e.g. explain: skipped-offline) but never changes the verdict.

3. Why does the review checklist "locate and follow" the installed explain skill instead of embedding its steps?

The explain skill lives in a separate, independently-evolving repo. Copying its ~500 lines into the checklist would silently drift; referencing it keeps one source of truth.

4. P8 added a narrow exception to the executor's "never commit" rule. What does it permit?

The carve-out is deliberately tiny: KB repo only (a different git root), offline fallback only, no push, and never in this workspace's repo — so the workspace's commit-ownership invariant is untouched.

5. Why were WebSearch/WebFetch added only to the Claude slice-executor-high, and why was that safe?

Reviews always run on slice-executor-high, so only it needs the tools for the cited best-practices section. They are read-only (no mutation/commit risk), and sync-agents only patches model:/effort:, so the hand-added tools: line survives sync and doesn't trip the drift check.

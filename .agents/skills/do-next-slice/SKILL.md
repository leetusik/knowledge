# do-next-slice

Run `python3 scripts/workflow.py next`, then read `AGENTS.md` (or `CLAUDE.md`), `docs/current/*.md` as needed, `docs/index.json`, `works/state.json`, `works/backlog.md`, the selected slice folder, and the phase's `phase.md` (the phase notebook тАФ accumulated decomposition, findings, and cross-slice notes). If you are ever unsure of the operator's intent, consult the phase's `intent.md` (linked from `phase.md`) тАФ the confirmed record of what was asked.

You are the ORCHESTRATOR (main thread): you plan each slice, verify, commit, move workflow state, and talk to the operator. The execution of every slice тАФ decomposition, implementation, `fix`, and the phase **review** тАФ is delegated to a `slice-executor` tier subagent (`slice-executor-low`, `-mid`, or `-high`; you do not do that slice's work yourself). One slice, then stop.

If `next` prints `WAITING ON OPERATOR` (the current slice or phase is `pending`, shown `[~]`), STOP: the work is waiting on operator co-work. Report what is needed and do not start, finish, or advance it. Resume only after the operator approves and clears the `pending` status back to `in_progress`.

Work exactly one slice:

1. If the selected slice is `todo` or `ready`, run `python3 scripts/workflow.py start-slice <slice_id>`. (In `plan only` mode, skip this тАФ the slice is readied, not started; see step 2.)
2. Plan the slice at the operator's gate before implementing. **Skip this step when the slice was selected as `ready` (`[r]`)** тАФ its `plan.md` was operator-approved by an earlier `plan only` pass, so re-read that `plan.md` and `phase.md` and dispatch the executor directly (step 3), re-planning through the gate only if the workspace state has visibly drifted from the plan. Otherwise: in Claude Code, **call the `EnterPlanMode` tool** to enter plan mode, do read-only research (surface any clarifying questions), then **call `ExitPlanMode`** to present the readied plan for the operator's approval. (In Codex, present the readied plan inline and wait for approval.) This is the slice-level intent step (refine тЖТ clarify тЖТ confirm when an operator note is ambiguous; consult `intent.md` when unsure). After approval (the harness exits plan mode), write the **operator-approved plan verbatim** to this slice's own `plan.md` тАФ persist the exact plan the operator just approved, in full, not a paraphrase or summary; free-form, no template; pull relevant context from `phase.md` and let the plan incorporate any operator note (the operator's verbatim intent lives in the phase's `intent.md`). Never pre-fill another slice's `plan.md`. If the operator invoked the skill with `auto` (e.g. `/do-next-slice auto`; "run unattended" also counts), do **not** enter plan mode тАФ plan inline, write `plan.md`, and dispatch the executor right away. `auto` waives only the approval gate, not the `pending` / `needs_operator` / `blocked` safety stops. **If the operator invoked the skill with `plan only`** (e.g. `/do-next-slice plan only`), ready this one slice's plan instead of executing it (if the selected slice is already `ready`, target the next slice in `order` that is neither `done` nor `ready`; if that is the `REVIEW` slice тАФ or nothing remains тАФ report and STOP: the review is never pre-planned, its plan depends on implementation results): run the plan gate exactly as above (`plan only` never combines with `auto` тАФ if both words appear, keep the approval gate), after approval write the approved plan verbatim to the slice's `plan.md`, mark it `python3 scripts/workflow.py set-slice-status <slice_id> ready` (shown `[r]` тАФ plan approved, awaiting execution), run `python3 scripts/workflow.py validate`, commit, report what was readied, and STOP тАФ no `start-slice`, no executor dispatch, no `finish-slice`, even for a decomposition slice (its readied plan executes on a later run).
3. Dispatch the executor to implement the slice. In Claude Code, always launch it via the Agent tool **as a background task** (the default тАФ never pass `run_in_background: false`) and then wait for its completion notification, doing nothing else in the meantime; never run the executor synchronously inline in the main session. This applies to every dispatch тАФ decomposition, implementation, `fix`, and the phase review. **Pick the tier by the slice's `kind` + `risk` (in `slice.json`):** decomposition and review slices always тЖТ **`slice-executor-high`**; an implementation/`fix` slice with `risk` exactly `low` тЖТ **`slice-executor-low`** (a literal plan-follower тАФ route to it only when `plan.md` is fully mechanical, with no judgment calls left); exactly `medium` тЖТ **`slice-executor-mid`**; anything else (`high`, unset, unknown) тЖТ **`slice-executor-high`**. You may bump a slice one tier UP (never down) when planning reveals complexity or the operator note says so; tier models/efforts come from the repo-root `executors.toml` + `sync-agents` (defaults haiku / sonnet@`xhigh` / opus@`xhigh`). Give the chosen subagent the slice id and folder path; it reads `plan.md`, `phase.md`, `slice.json`, the docs, and the code itself, implements, runs the slice's validation, writes `result.md`, appends durable cross-slice notes to `phase.md`, and returns a structured verdict (`status` = `done` | `needs_operator` | `blocked` | `escalate`, plus `summary`, `files_changed`, `validation`, `deviations`, `doc_impact`, and on `escalate` an `escalation` field with its findings; a review slice also returns `doc_versions` and a `review_verdict`). Do not implement the slice yourself. (In Codex, **spawn the matching tier** тАФ `.codex/agents/slice-executor-low.toml`, `-mid`, or `-high` by the same rule тАФ the same way; it does the slice's job against `plan.md` and returns the verdict; you never implement the slice inline.)
4. Trust the verdict; do not re-run the slice's work. Read the returned verdict and `result.md`, then run `python3 scripts/workflow.py validate` (state integrity only тАФ do **not** re-run the slice's tests; the phase review validates all slices together later). Then act on `status`:
   - `done` тЖТ continue to step 5.
   - `needs_operator` тЖТ set the slice `pending` (`python3 scripts/workflow.py set-slice-status <slice_id> pending`), report the `operator_need`, and STOP without finishing.
   - `blocked` тЖТ record the blocker in `result.md`, report it, and STOP.
   - `escalate` (returned only by `slice-executor-low`/`-mid`) тЖТ escalate one tier up instead of stopping: append an `## Escalation <n>: <from-tier> тЖТ <to-tier>` section to this slice's `plan.md` carrying the executor's `escalation` findings and what to do differently, then re-dispatch the next tier up (`low тЖТ mid тЖТ high`; jump straight to `high` only when the findings clearly show the slice was mis-rated). Present the revised plan for the operator's approval first (plan mode, as in step 2) unless running `auto`, where you re-dispatch straight away. At most 2 escalations per slice; the slice stays `in_progress` throughout.
   - failed or empty return тЖТ from `slice-executor-low`/`-mid`, treat it as an `escalate` with no findings (note the failed attempt in the escalation section and re-dispatch the next tier up); from `slice-executor-high`, treat the slice as not done: do not finish, do not commit; report and STOP.
5. Mark the slice done with `python3 scripts/workflow.py finish-slice <slice_id>` only when complete and verified.
6. Run `python3 scripts/workflow.py validate`.
7. Commit by default: group the slice's pending changes into focused `type(scope): summary` commit(s) following the Commit Convention. Committing is the orchestrator's job тАФ the executor never commits. Do not branch unless the operator asks; never push.

When the selected slice is a decomposition (`kind: decomposition`), it runs the **same path** as any slice тАФ you plan it, the `slice-executor` does its job. That job is to create the phase's middle slices: in step 3 the executor runs `python3 scripts/workflow.py new-slice --phase <P> --slice <P>.S<n> --name "..."` (with `--kind`, `--risk`, `--order`, `--depends-on` as the plan specifies) to create them as **bare folders** тАФ never pre-filling their `plan.md`; each fills its own when it runs тАФ and records the slice breakdown (what each slice covers and why) plus findings in the phase's `phase.md`, so later slices share that context. Set each middle slice's `--risk` deliberately тАФ `risk` selects the executor tier (`low` тЖТ `slice-executor-low`, `medium` тЖТ `slice-executor-mid`, anything else тЖТ `slice-executor-high`), so it is the phase's main cost lever; rate a slice `low` only when its work will be fully mechanical. Plan the decomposition accordingly; you still verify, `finish-slice`, and commit as for any slice.

When the selected slice is a phase review (`kind: review`), step 3 delegates the review to `slice-executor-high` like any slice тАФ you do not review it yourself. Plan it so the executor: validates all of the phase's slices together (each slice's validation commands from its `plan.md` / `result.md`, plus `python3 scripts/workflow.py validate`), reviews the phase against its objective, `intent.md`, and the docs, and тАФ **only on a passing review** тАФ consolidates the phase's durable-doc changes (the running "Doc impact" notes in `phase.md`) into new doc versions. On a review slice the executor writes only docs, never source code. (In Codex, spawn `slice-executor-high` from `.codex/agents/slice-executor-high.toml` the same way.) It returns a `review_verdict`; you record it and act on it instead of running steps 5тАУ7:

- Record the verdict: `python3 scripts/workflow.py review-phase <P> --verdict pass|changes_requested|blocked --reviewer slice-executor-high --note "..."`, then `python3 scripts/workflow.py validate` (state integrity тАФ also catches a stale `docs/current`).
- On `pass`: `review-phase` has already marked the phase `done` **and closed the `REVIEW` slice** тАФ no separate `finish-slice` is needed. The phase **stays in `active/`** тАФ archiving is a separate, manual step, so do **not** archive now. Archive later, when the operator asks: `archive-all` once every active phase is done, `rotate-backlog` to archive just the done phases while others continue, or `archive-phase <P>` for one phase.
- On `changes_requested`: `review-phase` returns the phase to `in_progress` and sets the `REVIEW` slice to `changes_requested` (kept open for re-review). Create the executor's proposed fix slices (`python3 scripts/workflow.py new-slice --phase <P> --slice <P>.F<n> --name "..." --kind fix`), complete them, then re-review; do not finish or archive. Docs stay unversioned until the eventual passing re-review consolidates them.
- On `blocked`: record the blocker; do not finish or archive.

Durable-doc versioning happens **only** at the phase review: the review-slice executor consolidates the phase's "Doc impact" notes (recorded in `phase.md` by earlier slices) into new versions on a passing review, and reports them in `doc_versions`; you confirm via `validate`. Implementation, `fix`, and decomposition slices never run `doc-new-version` тАФ when they change durable truth they append a one-line "Doc impact" note to `phase.md`. Never patch `docs/current/*.md` or old versions.

Stop after one slice. Do not advance to the next slice in the same turn.

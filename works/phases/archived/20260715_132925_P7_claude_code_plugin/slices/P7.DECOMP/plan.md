## Pinned research — treat as ground truth (verified against official docs + this repo on 2026-07-14)

### Claude Code plugin/marketplace format

- One repo can be both marketplace and plugin. Marketplace manifest: `.claude-plugin/marketplace.json` at repo root — `name` (kebab), `owner{name}`, `plugins[]` entries with `name` + `source`. Install flow: `/plugin marketplace add <owner>/<repo>` (or a local path for testing) → `/plugin install <plugin>@<marketplace>`.
- Plugin manifest: `<plugin-root>/.claude-plugin/plugin.json`; `name` required (kebab-case); recommended metadata: `version` (set it HERE ONLY — never also in the marketplace entry; installers receive updates only on version bumps), `description`, `author`, `homepage`, `license`, `keywords`.
- Components live at PLUGIN ROOT, never inside `.claude-plugin/`: `skills/<name>/SKILL.md` (invoked `/plugin-name:skill-name`, always namespaced; put `name:` in frontmatter), `commands/` (legacy — avoid), `agents/`, `hooks/`. Arbitrary payload dirs (e.g. `templates/`) are allowed and reachable at runtime via `${CLAUDE_PLUGIN_ROOT}` (usable inside skill bodies).
- The plugin's `source` directory is **copied whole** to every installer's `~/.claude/plugins/cache/`. `source: "./"` would ship the operator's personal `docs/`, `works/`, `data/` → the payload MUST live in a subdirectory (`plugin/`), marketplace entry `source: "./plugin"`. Path traversal (`..`) rejected; symlinks pointing outside the plugin are skipped — do not rely on symlinks for template sync (Windows git materializes them as text files).
- NO install-time setup hook exists. First-run setup = a skill the user runs (`/knowledge:setup`). `${CLAUDE_PLUGIN_DATA}` = per-plugin persistent dir (survives updates) — available but our config model deliberately does not depend on it.
- Dev tooling: `claude plugin validate <dir>` (add `--strict` for warnings-as-errors); local E2E via `/plugin marketplace add ./` or `claude --plugin-dir`.
- A CLAUDE.md inside the plugin is NOT loaded as context. Reserved marketplace names exist (claude-plugins-official etc.) — `knowledge` is fine.

Three file classes: **byte-identical** (server/*, scripts/graph_hook.py, scripts/site_smoke.py after de-hardcoding, Dockerfile, pyproject.toml, uv.lock, .dockerignore, pages.yml, docs/graph.md, docs/tags.md, docs/assets|stylesheets|javascripts, tests/*), **parameterized-at-scaffold** (mkdocs.yml site_name/site_url/copyright; compose.yml TZ/ports), **template-only** (generic docs/index.md with marker + seed bullet + graph card, one seed welcome explainer with valid frontmatter so graph_hook emits a node, generic Makefile, .gitignore incl. .env). Runtime config stays runtime env (config.py already handles it).
3. **Config model (SaaS-open)**: setup writes `~/.config/knowledge-kb/config.json` (honor `$XDG_CONFIG_HOME`; keys kb_root, api.base_url, api.token, site.base_url; chmod 600). Shipped explain resolution order: env (`KB_API_BASE_URL`/`KB_ROOT`/`KB_API_TOKEN`) → config file → legacy `~/projects/personal/knowledge` convention (keeps the operator's machines working pre-setup) → STOP "run /knowledge:setup". Bearer header when token configured. File+git fallback allowed ONLY when config resolves a LOCAL kb_root; remote base_url unreachable → report, never fallback.
4. **Setup skill UX**: asks target dir (default `~/knowledge`), site title, GitHub owner/repo (optional → site_url; skipped = local-only), TZ (host default), ports (defaults 8765/8766), optional Gemini key (never written into the scaffold — host env only). Does: render.py scaffold + `.kb-scaffold.json` marker (records plugin/template version + params) → git init + initial commit → write config file → if Docker: `docker compose up -d` + healthz/site probe, else print uv alternative (`uv run uvicorn server.main:app --port <port>`) → print GitHub Pages enablement steps + verify checklist. Idempotent: marker present → reconfigure-only / re-render-with-diff-and-confirm / abort; non-empty dir without marker → refuse. Degraded modes: no Docker (file-fallback KB still fully works), no GitHub (local-only), no Gemini (BM25-only).
5. **LICENSE**: MIT (operator decision 2026-07-14) — root LICENSE + `license: "MIT"` in plugin.json.
6. **Scope guards**: Claude Code plugin only (the `.agents/` Codex mirror is the bootstrap workspace's concern — out of scope); bootstrap repo untouched; never push; release discipline = any `plugin/**` change pairs with a plugin.json version bump (document in a release checklist); secrets hygiene (no token/key in scaffolds; config 600).

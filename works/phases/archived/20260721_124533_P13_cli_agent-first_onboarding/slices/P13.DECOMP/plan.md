# P13.DECOMP — Plan (decompose the CLI & Agent-First Onboarding phase)

Orchestrator's native plan for the **`P13.DECOMP`** slice, executed by **`slice-executor-high`**. The executor's only job is to create P13's middle slices as **bare folders** and seed the phase notebook — **no CLI code, no slice `plan.md`, no `doc-new-version`, no commits.** Per-slice implementation plans are written later, one at a time, at each slice's own turn.

## Context

P13 is **"let's say you are my user"**: a person inside Claude Code or Codex must run the whole lifecycle — sign up, log in, configure credentials, save/search knowledge — **without ever visiting the website**. The deliverable is a **standalone installable CLI** (explicitly *not* a plugin feature) plus **agent-readable guide docs** so a coding agent can drive the flow on the user's behalf. The existing `/knowledge:explain` + `/knowledge:setup` plugin **stays untouched** as the self-host open-core path.

Research verified against the tree at 2026-07-17. Six facts are load-bearing for the breakdown:

**1. The whole onboarding sequence already exists in Python.** `scripts/onboarding_smoke.py` (326 lines) drives signup → project → credential → write → search → list → usage over `httpx`, and is the committed post-deploy verifier. The CLI is largely a *productization* of this proven sequence, not a discovery exercise. Its `FROZEN_201_KEYS` (`:51-55`) pin the write contract.

**2. The config seam is real but has never been implemented in code.** `$XDG_CONFIG_HOME/knowledge-kb/config.json` (default `~/.config/knowledge-kb/config.json`) exists **only as prose + inline `python3 -c` heredocs** in two SKILL.md files — a repo-wide grep for `knowledge-kb` across `*.py`/`*.ts`/`*.sh` returns **zero code hits**. The nested schema is fixed at `plugin/skills/setup/SKILL.md:192-201` (`kb_root`, `api.base_url`, `api.token`, `site.base_url`); the sole reader is the resolver at `plugin/skills/explain/SKILL.md:31-78` ("a flat shape will not be read"). **The CLI is the first code to touch this seam** and must match the prose resolver exactly.

**This is the phase's payoff.** `setup` always writes `token: null` and "never writes a bearer token" (`setup/SKILL.md:203`) — nothing today ever populates it. A `vk_` key exists only in an HTTP response. When the CLI writes `api.base_url=https://knowledge.hi2vi.com` + `api.token=vk_…` into that file, **`/knowledge:explain` starts writing to the hosted SaaS with zero code change** — proving "SaaS-open" (`docs/versions/security/v0003_*.md:42-50`) was real. `kb_root` may legitimately be absent (remote-only config), and the local-fallback rule (`explain/SKILL.md:216-220`) already guarantees a remote KB can never silently degrade into a stray local write.

**3. The production edge does not route the control plane — the CLI cannot work against the deployed box today.** `deploy/knowledge.conf:134-165` proxies **only** `location /api/` and `location = /healthz` to the API; `location /` catches everything else and goes to the mkdocs site. **`/auth/*` and `/app/*` fall through to the static site and 404** — deliberately ("the frozen consumer contract is `/api/*` + `/healthz` only", `:136-139`). The web app is also absent from `compose.prod.yml` (services: `api`, `site`, `postgres`). So signup/login/project/credential — the entire first half of the CLI's job — is unreachable at `https://knowledge.hi2vi.com`. **Without an edge change P13's intent cannot be met**, so this phase exposes the control plane (§S5); P14 later adds the web-app vhost + landing on top.

**4. Exposing `/auth/*` publicly is a security event.** There is **no server-side rate limiting anywhere** — the only throttle is in the Next BFF (`web/src/app/api/auth/login/route.ts:17-18`, 5/IP/15min), which a direct API call **bypasses entirely**. `POST /auth/login` is a plain JSON password grant with no CSRF/cookie/origin check. Publishing it at the edge without a throttle opens unmetered password-guessing and open signup to the internet. S5 must land throttling **in the same slice** as the exposure — never exposure alone.

**5. The repo is deliberately not installable.** Root `pyproject.toml` has **no `[build-system]`** and `[tool.uv] package = false` (`:24-29`) — a *virtual* project, deps-only, because `server/` runs bind-mounted from the repo root and Docker reproduces it with `uv export --no-emit-project`. There is no `src/` layout and **no `[project.scripts]` anywhere in the repo**. So the CLI **cannot** hang off the root package: it needs its own installable package. vocky's packaging is the shape to mirror — hatchling + `[project.scripts]` + `packages = ["src/vocky"]` — but in a **subdirectory**, leaving the root virtual project untouched.

**6. The vocky reference is thinner than the intent assumes — there is no CLI to copy.** vocky's P5 ("CLI & Agent-First Onboarding", the phase P13 mirrors) is a **`planned` stub that was never decomposed**: two slice stubs (`P5.DECOMP`, `P5.REVIEW`, both `todo`), an empty `phase.md`, no `plan.md`/`result.md`, zero P5 commits. It records **the same three open questions** (device-flow vs direct credentials, distribution channel, guide-doc location) and **answers none** — vocky framed the questions, it did not resolve them. Beyond that: `grep -rniE "typer|click|argparse" src/` → **zero**; no cli module; **no client config-file convention at all** (no `~/.vocky`, no XDG, no keyring — its only config is server-side pydantic-settings). So P13 is **greenfield**, and on the client-config question **knowledge is ahead of vocky**: the `knowledge-kb` seam already exists as a designed contract. What vocky *is* good for: (a) the onboarding sequence in `src/vocky/smoke.py` — the identical signup → project → `vk_` walk knowledge already mirrors in `onboarding_smoke.py`; (b) the API shape (same three-surface `/auth` `/app` `/api` split, same 30-day session TTL, same show-once `vk_`, same 404-not-403, same generic-401 — knowledge's control plane was ported from it); (c) `docs/current/experience.md`, its "an agent could drive this" onboarding narrative, a **tone** reference for S4.

**A useful negative signal for S4:** vocky *does* serve agent-readable docs from the API (`GET /admin/assets/docs.json`, packaged into the wheel, `importlib.resources` + `lru_cache`) — and is **retiring that pattern**. Per its D-P2-4 the packaged docs went stale, were superseded by the web app rather than updated, and get deleted with the Admin UI in its P6. So the "served by the API" option for guide docs has a real, observed failure mode next door: served docs drift and rot. That is evidence for bundling them with the CLI (versioned with the code), not against it.

## Slice breakdown to create (5 middle slices)

Foundation **S1**, then the two command surfaces **S2 → S3** (S3 needs S2's auth), then **S4** (docs describe the shipped commands) and **S5** (the deploy/exposure gate + E2E). Two `high` (credential handling; public auth exposure), three `medium`, none `low` — nothing here is fully mechanical.

- **P13.S1 — CLI package + config seam + API client** · `implementation` · `medium` · order 1
  The installable foundation. New **`cli/` subdirectory package** with its **own** `pyproject.toml` (hatchling, `[project.scripts]`, `packages = ["src/knowledge_cli"]`), leaving the root virtual project **untouched** — so `uv tool install git+https://github.com/leetusik/knowledge#subdirectory=cli` works with **no PyPI account**. Three modules: (a) a **config-seam module** — the first *code* implementation of `~/.config/knowledge-kb/config.json`, byte-compatible with the `explain/SKILL.md:31-78` resolver (nested schema, per-key env precedence `KB_ROOT`/`KB_API_BASE_URL`/`KB_API_TOKEN`, atomic write, **`chmod 600` actually enforced** — today it is only a prose instruction nothing checks); (b) a typed **HTTP client** over `/auth` + `/app` + `/api` (`httpx`, bearer injection, an `ApiError` that never echoes a raw body), lifted from `onboarding_smoke.py`'s proven calls; (c) the command skeleton + `--help`. **CLI tests live under `cli/tests/`, never root `tests/`** (parity guard). No behavior yet.
  _Rationale:_ packaging + a seam that must match an existing prose contract exactly → `medium`, not `low`.

- **P13.S2 — Auth & onboarding commands** · `implementation` · `high` · order 2 · depends P13.S1
  The credential-handling core and the phase's headline flow. `signup`, `login`, `logout`, `whoami`, and a one-shot **`init`/`onboard`** that runs the whole `onboarding_smoke.py` sequence for a real user: signup-or-login → create project → mint `vk_` → **write the config seam** (`api.base_url` + `api.token=vk_…`, `chmod 600`) → verify. Password prompts must never echo, never land in argv (shell history) or logs; the **show-once `vk_`** is written to config and never re-printed/logged. Session token (30-day TTL, `auth_api.py:42`) stored under a **new additive key** (`auth.session_token`) — extra keys are ignored by the existing resolver, so this stays backward-compatible; `api.token` gets the **non-expiring `vk_`**, which is what makes `/knowledge:explain` keep working after the session lapses. Handle signup **409** (duplicate email → offer login) and the generic **401** (`login` never distinguishes bad-email from bad-password — enumeration-safety, preserve it).
  _Rationale:_ secret handling + the file that lights up every other client → top tier.

- **P13.S3 — Knowledge commands** · `implementation` · `medium` · order 3 · depends P13.S2
  What the user actually does day to day: `save` (`POST /api/documents` — enforce the **2–5 tags** rule (`documents.py:61-62`) *client-side* with a clear message rather than surfacing a raw 422; handle **409** duplicate → `--overwrite`; markdown is the body **without** frontmatter, starting at the H1), `search`, `list`, `read`, `projects`, `usage`. Reads go through `vk_` on `/api/*`. Note `body.project` is a free-form **name string** never checked against the key's bound project (`main.py:565-571`) — the CLI should default it from config rather than let a user drift.
  _Rationale:_ several endpoints + real error-shape mapping, all against a frozen contract → `medium`.

- **P13.S4 — Agent-readable guide docs + discovery** · `implementation` · `medium` · order 4 · depends P13.S3
  The half of the intent that is not code: docs written **for a coding agent**, not a human skimmer, so Claude Code/Codex can drive the whole flow unattended. Decide **where they live and how agents discover them** (the intent's explicit open question: bundled with the CLI? served by the API? both?) — the cheap, discoverable answer is a `knowledge guide` command emitting agent-oriented markdown **bundled in the package** (works offline, versioned with the code), plus a short `plugin/README.md`-style human quickstart. **Vocky's retirement of its served `docs.json` (its D-P2-4: packaged docs went stale, superseded rather than updated, deleted in its P6) is the evidence for bundling over serving.** Must cover the full lifecycle incl. the non-obvious constraints (2–5 tags, show-once `vk_`, remote-only config, no local fallback). Tone reference: `~/projects/personal/vocky/docs/current/experience.md`. **No new `server/` route unless S4 chooses "served by the API"** — which would add parity debt (§shipped_dirs) and needs a deliberate call at its planning turn.
  _Rationale:_ a real design decision (discovery mechanism) + prose that must be exactly right → `medium`.

- **P13.S5 — Expose the control plane at the edge + throttle `/auth` + E2E CLI smoke** · `implementation` · `high` · order 5 · depends P13.S3
  What makes the phase **real** instead of local-only. Add `location /auth/` + `location /app/` to `deploy/knowledge.conf` (both currently 404 into the mkdocs site), **and in the same slice** land throttling for the now-public password grant — nginx `limit_req` and/or a server-side limiter — because the BFF's 5/IP/15min is bypassed by direct API calls. Revisit the "frozen consumer contract is `/api/* + /healthz` only" comment (`:136-139`) honestly: this widens it, and `docs/current/api.md` must say so. Then an **E2E CLI smoke** in the `onboarding_smoke.py` style (argparse, collect-all-failures, PASS/exit-1) driving the *installed* CLI end to end, and the operator-facing verification that `/knowledge:explain` writes to the SaaS through the CLI-written config.
  _Rationale:_ publishing an unthrottled auth surface to the internet is the riskiest thing in the phase → top tier. Operator was told this slice widens the public contract and approved the plan on that basis.

**Why 5:** the intent has two explicit deliverables (CLI + agent docs) but the CLI is three separable concerns (package/seam foundation, auth+credential flow, knowledge commands) that differ sharply in risk — merging S2's secret handling into S1's packaging would drag the whole foundation to `high`. S5 exists because research showed the deployed box **cannot** serve the CLI at all today; without it P13 ships something that only works on localhost, which is not what "a user doesn't even need to visit our website" means.

## Decisions to record (Findings & Notes → Decisions)

- **D-P13-1 — CLI is a separate package in `cli/`.** Own `pyproject.toml` (hatchling + `[project.scripts]`), **root virtual project untouched** (`package = false` is load-bearing for Docker's `uv export --no-emit-project`). Distribution: `uv tool install git+https://github.com/leetusik/knowledge#subdirectory=cli` — **no PyPI account, no npm, no brew**. PyPI stays a later, separate call.
- **D-P13-2 — Auth flow: direct password grant, not device flow.** `POST /auth/login` is already the programmatic path (plain JSON, bearer out, no CSRF/cookie/redirect); **no device flow exists in either repo** and building one means net-new server endpoints. The intent leaves the choice to DECOMP; vocky left the same question open and never answered it, so there is no precedent to inherit — P13 decides it. Cheap answer wins: use what exists. Revisit only if the operator wants browser-based SSO.
- **D-P13-3 — Two tokens, two lifetimes.** Session token (30d) → new additive `auth.session_token` key, used for `/app/*`. **`vk_` (non-expiring) → `api.token`**, the seam `/knowledge:explain` already reads. This is what makes the plugin keep working after the session expires, and is the concrete proof of "SaaS-open".
- **D-P13-4 — The control plane goes public in P13, with a throttle, or the phase is localhost-only.** Exposure and rate limiting land together in S5, never apart.
- **D-P13-5 — CLI code and CLI tests stay out of `server/` and `tests/`.** `shipped_dirs` makes those dirs parity-guarded; the guard is already failing and P13 adds no new debt to it.
- **D-P13-6 — Guide docs ship bundled in the CLI package, not served by the API.** Offline, versioned with the code, no parity debt. Vocky's served-`docs.json` rotted and is being deleted (its D-P2-4) — that is the observed failure mode being avoided. (S4 may overturn this at its planning turn with a reason.)
- **Untouched by decree (intent):** `/knowledge:explain` + `/knowledge:setup` and `plugin/` remain the self-host open-core path. The CLI *writes the config they read*; it does not modify them.

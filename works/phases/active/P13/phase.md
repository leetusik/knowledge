# Phase P13: CLI & Agent-First Onboarding

_Intent: see [intent.md](intent.md)._

## Objective

Standalone installable CLI so a user inside Claude Code or Codex can sign up, log in, configure credentials, and use the knowledge features end to end, with agent-readable help/guide docs so the website is never required

## Context

P13 is **"let's say you are my user"**: a person working inside Claude Code or Codex must run the whole lifecycle ‚Äî sign up, log in, configure credentials, save/search knowledge ‚Äî **without ever visiting the website**. The deliverable is a **standalone installable CLI** (explicitly *not* a plugin feature) plus **agent-readable guide docs** so a coding agent can drive the flow on the user's behalf. The existing `/knowledge:explain` + `/knowledge:setup` plugin **stays untouched** as the self-host open-core path.

Five facts shape everything below (all re-verified against the tree at 2026-07-17 by `P13.DECOMP`):

**1. The CLI is a productization, not a discovery exercise.** `scripts/onboarding_smoke.py` (326 lines) already drives the entire sequence ‚Äî signup ‚Üí project ‚Üí credential ‚Üí write ‚Üí search ‚Üí list ‚Üí usage ‚Äî over `httpx`, and is the committed post-deploy verifier. The CLI re-packages a *proven* sequence. Its `FROZEN_201_KEYS` (`:51-55`) pin the write contract.

**2. The config seam is real prose but has never been code ‚Äî and that is the phase's payoff.** `$XDG_CONFIG_HOME/knowledge-kb/config.json` (default `~/.config/knowledge-kb/config.json`) exists **only** as prose + inline `python3 -c` heredocs in two SKILL.md files; a repo-wide grep for `knowledge-kb` across `*.py`/`*.ts`/`*.sh` returns **zero code hits**. The CLI is the **first code to touch this seam** and must match the prose resolver byte-for-byte (nested schema ‚Äî "a flat shape will not be read"). Today `setup` always writes `token: null` and "never writes a bearer token" ‚Äî *nothing populates it*, and a `vk_` key exists only in an HTTP response. When the CLI writes `api.base_url=https://knowledge.hi2vi.com` + `api.token=vk_‚Ä¶` into that file, **`/knowledge:explain` starts writing to the hosted SaaS with zero code change** ‚Äî proving "SaaS-open" (`docs/versions/security/v0003_*.md:42-50`) was real. `kb_root` may legitimately be absent (remote-only config); the local-fallback STOP rule already guarantees a remote KB can never silently degrade into a stray local write.

**3. The deployed box cannot serve the CLI today ‚Äî the control plane is unrouted.** `deploy/knowledge.conf:134-165` proxies **only** `location /api/` and `location = /healthz`; `location /` catches everything else into the mkdocs site. So **`/auth/*` and `/app/*` 404 into the static site** ‚Äî deliberately ("the frozen consumer contract is `/api/*` + `/healthz` only", `:136-139`). The web app is also absent from `compose.prod.yml` (services: `api`, `site`, `postgres` only). Signup/login/project/credential ‚Äî the CLI's entire first half ‚Äî is unreachable at `https://knowledge.hi2vi.com`. **Without an edge change P13's intent cannot be met** ‚Üí S5. P14 later adds the web-app vhost + landing on top.

**4. Exposing `/auth/*` is a security event.** There is **no server-side rate limiting anywhere**; the only throttle is in the Next BFF (`web/src/app/api/auth/login/route.ts`, 5/IP/15min), which a direct API call **bypasses entirely**. `POST /auth/login` is a plain JSON password grant with no CSRF/cookie/origin check. Publishing it without a throttle opens unmetered password-guessing and open signup to the internet ‚Üí exposure and throttling land **in the same slice** (S5), never apart.

**5. The repo is deliberately not installable, so the CLI needs its own package.** Root `pyproject.toml` has **no `[build-system]`** and `[tool.uv] package = false` ‚Äî a *virtual* project, deps-only, because `server/` runs bind-mounted from the repo root and Docker reproduces it with `uv export --no-emit-project`. No `src/` layout, and **no `[project.scripts]` anywhere in the repo**. The CLI cannot hang off the root package ‚Üí its own `cli/` subdirectory package, root untouched.

**Vocky is reference-only ‚Äî there is no CLI to copy.** Vocky's P5 (the phase P13 mirrors) is a `planned` stub that was never decomposed: two `todo` slice stubs, empty `phase.md`, no plans, zero P5 commits. It records **the same three open questions** (device-flow vs direct credentials, distribution channel, guide-doc location) and **answers none**. `grep -rniE "typer|click|argparse" src/` ‚Üí zero Python hits; no cli module; **no client config-file convention at all**. So P13 is greenfield, and on client config **knowledge is ahead of vocky**. What vocky *is* good for: (a) `src/vocky/smoke.py`, the identical onboarding walk; (b) the API shape (same `/auth` `/app` `/api` split, 30-day TTL, show-once `vk_`, 404-not-403, generic-401 ‚Äî knowledge's control plane was ported from it); (c) `docs/current/experience.md` as a **tone** reference for S4's agent-narrative docs; (d) `pyproject.toml:36-42` (hatchling + `[project.scripts]` + `packages = ["src/vocky"]`) as the packaging shape.

## Decomposition

Five middle slices. Foundation **S1**, then the two command surfaces **S2 ‚Üí S3** (S3 needs S2's auth), then **S4** (docs describe the shipped commands) and **S5** (the deploy/exposure gate + E2E). Two `high` (credential handling; public auth exposure), three `medium`, **none `low`** ‚Äî nothing here is fully mechanical.

- **P13.S1 ‚Äî CLI package + config seam + API client** ¬∑ `implementation` ¬∑ `medium` ¬∑ order 1
  The installable foundation. New **`cli/` subdirectory package** with its **own** `pyproject.toml` (hatchling, `[project.scripts]`, `packages = ["src/knowledge_cli"]`), leaving the root virtual project **untouched** ‚Äî so `uv tool install git+https://github.com/leetusik/knowledge#subdirectory=cli` works with **no PyPI account**. Three modules: (a) a **config-seam module** ‚Äî the first *code* implementation of `~/.config/knowledge-kb/config.json`, byte-compatible with the `explain/SKILL.md:31-78` resolver (nested schema, per-key env precedence `KB_ROOT`/`KB_API_BASE_URL`/`KB_API_TOKEN`, atomic write, **`chmod 600` actually enforced** ‚Äî today it is only a prose instruction nothing checks); (b) a typed **HTTP client** over `/auth` + `/app` + `/api` (`httpx`, bearer injection, an `ApiError` that never echoes a raw body), lifted from `onboarding_smoke.py`'s proven calls; (c) the command skeleton + `--help`. **CLI tests live under `cli/tests/`, never root `tests/`** (parity guard). No behavior yet.
  _Rationale:_ packaging + a seam that must match an existing prose contract exactly ‚Üí `medium`, not `low`.

- **P13.S2 ‚Äî Auth & onboarding commands** ¬∑ `implementation` ¬∑ `high` ¬∑ order 2 ¬∑ depends P13.S1
  The credential-handling core and the phase's headline flow. `signup`, `login`, `logout`, `whoami`, and a one-shot **`init`/`onboard`** that runs the whole `onboarding_smoke.py` sequence for a real user: signup-or-login ‚Üí create project ‚Üí mint `vk_` ‚Üí **write the config seam** (`api.base_url` + `api.token=vk_‚Ä¶`, `chmod 600`) ‚Üí verify. Password prompts must never echo, never land in argv (shell history) or logs; the **show-once `vk_`** is written to config and never re-printed/logged. Session token (30-day TTL, `auth_api.py:42`) stored under a **new additive key** (`auth.session_token`) ‚Äî extra keys are ignored by the existing resolver, so this stays backward-compatible; `api.token` gets the **non-expiring `vk_`**, which is what makes `/knowledge:explain` keep working after the session lapses. Handle signup **409** (duplicate email ‚Üí offer login) and the generic **401** (`login` never distinguishes bad-email from bad-password ‚Äî enumeration-safety, preserve it).
  _Rationale:_ secret handling + the file that lights up every other client ‚Üí top tier.

- **P13.S3 ‚Äî Knowledge commands** ¬∑ `implementation` ¬∑ `medium` ¬∑ order 3 ¬∑ depends P13.S2
  What the user actually does day to day: `save` (`POST /api/documents` ‚Äî enforce the **2‚Äì5 tags** rule (`documents.py:61-62`) *client-side* with a clear message rather than surfacing a raw 422; handle **409** duplicate ‚Üí `--overwrite`; markdown is the body **without** frontmatter, starting at the H1), `search`, `list`, `read`, `projects`, `usage`. Reads go through `vk_` on `/api/*`. Note `body.project` is a free-form **name string** passed only through `validate_project()` and never checked against the key's bound project (`main.py:396`, hint at `:565-571`) ‚Äî the CLI should default it from config rather than let a user drift.
  _Rationale:_ several endpoints + real error-shape mapping, all against a frozen contract ‚Üí `medium`.

- **P13.S4 ‚Äî Agent-readable guide docs + discovery** ¬∑ `implementation` ¬∑ `medium` ¬∑ order 4 ¬∑ depends P13.S3
  The half of the intent that is not code: docs written **for a coding agent**, not a human skimmer, so Claude Code/Codex can drive the whole flow unattended. Decide **where they live and how agents discover them** (the intent's explicit open question: bundled with the CLI? served by the API? both?) ‚Äî the cheap, discoverable answer is a `knowledge guide` command emitting agent-oriented markdown **bundled in the package** (works offline, versioned with the code), plus a short `plugin/README.md`-style human quickstart. **Vocky's retirement of its served `docs.json` (its D-P2-4: packaged docs went stale, were superseded rather than updated, and are deleted with the Admin UI in its P6) is the evidence for bundling over serving** ‚Äî a real, observed drift-and-rot failure mode next door. Must cover the full lifecycle incl. the non-obvious constraints (2‚Äì5 tags, show-once `vk_`, remote-only config, no local fallback). Tone reference: `~/projects/personal/vocky/docs/current/experience.md`. **No new `server/` route unless S4 chooses "served by the API"** ‚Äî which would add parity debt (¬ß`shipped_dirs`) and needs a deliberate call at its planning turn.
  _Rationale:_ a real design decision (discovery mechanism) + prose that must be exactly right ‚Üí `medium`.

- **P13.S5 ‚Äî Expose the control plane at the edge + throttle `/auth` + E2E CLI smoke** ¬∑ `implementation` ¬∑ `high` ¬∑ order 5 ¬∑ depends P13.S3
  What makes the phase **real** instead of local-only. Add `location /auth/` + `location /app/` to `deploy/knowledge.conf` (both currently 404 into the mkdocs site), **and in the same slice** land throttling for the now-public password grant ‚Äî because the BFF's 5/IP/15min is bypassed by direct API calls. **The nginx route is not free: `deploy/knowledge.conf:32,35-40` bans `limit_req_zone` in this vhost** ‚Äî zone names are global across the `conf.d/` tree, `hi2vi.conf` owns the tree's only one (`hi2vi_contact`), and a duplicate name is a hard `nginx -t` failure that blocks the reload for **every site on the edge**. So S5 must either add a uniquely-named zone with the house rules re-read in full, or land the throttle **server-side** ‚Äî a deliberate call at its planning turn. Cloudflare real-IP restore is already in place (`:83`, "any future limiter" keys off real visitors). Revisit the "frozen consumer contract is `/api/* + /healthz` only" comment (`:136-139`) honestly: this widens it, and `docs/current/api.md` must say so. Then an **E2E CLI smoke** in the `onboarding_smoke.py` style (argparse, collect-all-failures, PASS/exit-1) driving the *installed* CLI end to end, and the operator-facing verification that `/knowledge:explain` writes to the SaaS through the CLI-written config.
  _Rationale:_ publishing an unthrottled auth surface to the internet is the riskiest thing in the phase ‚Üí top tier. Operator was told this slice widens the public contract and approved the plan on that basis.

**Why 5:** the intent has two explicit deliverables (CLI + agent docs) but the CLI is three separable concerns (package/seam foundation, auth+credential flow, knowledge commands) that differ sharply in risk ‚Äî merging S2's secret handling into S1's packaging would drag the whole foundation to `high`. S5 exists because research showed the deployed box **cannot** serve the CLI at all today; without it P13 ships something that only works on localhost, which is not what "a user doesn't even need to visit our website" means.

## Findings & Notes

### Decisions

- **D-P13-1 ‚Äî CLI is a separate package in `cli/`.** Own `pyproject.toml` (hatchling + `[project.scripts]`), **root virtual project untouched** (`package = false` is load-bearing for Docker's `uv export --no-emit-project`). Distribution: `uv tool install git+https://github.com/leetusik/knowledge#subdirectory=cli` ‚Äî **no PyPI account, no npm, no brew**. PyPI stays a later, separate call. _(Resolves the intent's open "distribution channel" question.)_
- **D-P13-2 ‚Äî Auth flow: direct password grant, not device flow.** `POST /auth/login` is already the programmatic path (plain JSON, bearer out, no CSRF/cookie/redirect); **no device flow exists in either repo** and building one means net-new server endpoints. The intent leaves the choice to DECOMP; vocky left the same question open and never answered it, so there is no precedent to inherit ‚Äî P13 decides it. Cheap answer wins: use what exists. Revisit only if the operator wants browser-based SSO.
- **D-P13-3 ‚Äî Two tokens, two lifetimes.** Session token (30d) ‚Üí new additive `auth.session_token` key, used for `/app/*`. **`vk_` (non-expiring) ‚Üí `api.token`**, the seam `/knowledge:explain` already reads. This is what makes the plugin keep working after the session expires, and is the concrete proof of "SaaS-open".
- **D-P13-4 ‚Äî The control plane goes public in P13, with a throttle, or the phase is localhost-only.** Exposure and rate limiting land together in S5, never apart.
- **D-P13-5 ‚Äî CLI code and CLI tests stay out of `server/` and `tests/`.** `shipped_dirs` makes those dirs parity-guarded; the guard is already failing and P13 adds no new debt to it.
- **D-P13-6 ‚Äî Guide docs ship bundled in the CLI package, not served by the API.** Offline, versioned with the code, no parity debt. Vocky's served-`docs.json` rotted and is being deleted (its D-P2-4) ‚Äî that is the observed failure mode being avoided. (S4 may overturn this at its planning turn with a reason.)
- **Untouched by decree (intent):** `/knowledge:explain` + `/knowledge:setup` and `plugin/` remain the self-host open-core path. The CLI *writes the config they read*; it does not modify them.

## Constraints

- Root `pyproject.toml` stays `package = false` with **no `[build-system]`** ‚Äî load-bearing for Docker's `uv export --no-emit-project`. The CLI never changes it.
- **CLI code and CLI tests out of `server/` and `tests/`** (parity-guarded `shipped_dirs`; the guard is already failing ‚Äî add no new debt).
- **`plugin/` + both SKILL.md files are untouched by decree** (intent). The CLI *writes the config they read*; it does not modify them.
- The config seam must match the existing prose resolver **exactly** ‚Äî nested schema only ("a flat shape will not be read"); new keys must be additive.
- Tags **2‚Äì5** on every write; the `vk_` key is **shown once**; `markdown` is the body without frontmatter, starting at the H1.
- **Generic-401 enumeration-safety preserved** ‚Äî `login` must never distinguish bad-email from bad-password.
- `deploy/knowledge.conf` house rules: **no `default_server`, no IPv6 `listen`, no `limit_req_zone`** (global zone names; a duplicate breaks every site on the edge). The `conf.d/` tree is tested and reloaded **as a unit**.
- Single uvicorn worker invariant unaffected.

### S2 landed â€” interfaces & gotchas for S3

- **App**: `server.main:app` (FastAPI). Reads open a **fresh per-request connection** via the `get_conn()` dependency (`db.connect()` in a `try/finally` that closes it) â€” config stays env-at-call-time, so tests retarget `KB_ROOT`/`KB_DB_PATH` per test. Nothing cached at import.
- **Bearer dependency to reuse (S3)**: `require_bearer(authorization: Optional[str] = Header(...))` in `server/main.py`. No-op when `config.api_token()` is None; else requires an exact `Authorization: Bearer <token>` header, else `HTTPException(401)`. **S3 adds it to `POST /api/documents`** via `Depends(require_bearer)` â€” do not reinvent. Today it guards only `POST /api/reindex`; GETs are always open.
- **Search API**: `server.search.search(conn, q, *, project=None, tag=None, limit=10, raw=False) -> list[dict]`; blank `q` â†’ `[]`; each result = doc fields (id/project/slug/date/title/tags list/rel_path/source_repo/created_at/updated_at) **minus markdown & tags_text** + `score` (=`-bm25`, higher-is-better, `round(...,4)`), `snippet` (`<mark>`-wrapped, col 2 = markdown), `signals: {"bm25": score}`. Helper `build_match_query(q)` quotes each whitespace token (doubling internal `"`) so raw FTS5 operators can never 500. `raw=True` passes `q` verbatim; a `sqlite3.OperationalError` is re-raised as `search.SearchQueryError`, which `main.py` maps to **HTTP 400**. Clean sqlite-vec/RRF fusion seam left in `search.py`.
- **Response shaping helper**: `_public_doc(doc, *, include_markdown)` in `main.py` always drops the internal `tags_text` FTS-mirror (redundant with the `tags` list) and drops `markdown` for list payloads. List omits markdown+tags_text; get-by-id/by-path include markdown (still no tags_text).
- **Route ordering gotcha**: `/api/documents/by-path/{rel_path:path}` is declared **before** `/api/documents/{doc_id}` (and `doc_id` is typed `int`), so by-path never binds to the id route. S3's `POST /api/documents` (no path param) is unaffected.
- **Reindex endpoint**: `POST /api/reindex` calls `reindex.reindex()` bare (it opens/closes its own connection, walks `config.docs_root()`, never runs git) and returns its `{indexed, removed, skipped, duration_ms}` dict as-is.
- **BM25 tiny-corpus note (not a bug)**: FTS5 bm25 collapses toward 0 when a term's IDF is ~0 â€” i.e. the term appears in every doc, or the corpus is 1â€“2 docs. On the **real 1-doc repo, `score` rounds to `0.0`** while the `<mark>` snippet + result still return correctly. The read-API tests therefore seed a **3-doc** temp corpus (term in 1 of 3) so `score > 0` is a meaningful assertion. Don't treat `score == 0.0` on the single-doc real repo as a regression.

## Discovered consideration â€” non-explainer files now inside `docs/`

The approved plan predates the agentic-workspace install and assumed `docs/` held only explainers. It now **also** holds workspace internals inside the MkDocs content root:

- `docs/current/*.md` (11 generated fullstack docs), `docs/versions/**/*.md` (versioned durable docs), `docs/README.md`, `docs/index.json` â€” none of these carry explainer frontmatter (title/date/tags/source).

**Reindex must handle them deliberately** so its output stays clean. The plan's reindex already skips top-level `index.md`/`tags.md`; **S1 decides** the mechanism for the rest â€” either explicitly skip the `current/` and `versions/` directories (and top-level non-explainer files like `README.md`, `index.json`), or let frontmatter validation route them into the `skipped[]` list with a reason. Either is acceptable; the requirement is that `indexed` counts only real explainers and `skipped[]` is not noisy garbage. Record the chosen mechanism in this notebook when S1 lands.

**S1 decision (landed):** chose the **hybrid** â€” (a) the walk only descends into `docs/<subdir>/` directories, so all top-level files (`index.md`, `tags.md`, `README.md`, `index.json`) never enter it; (b) a module constant `RESERVED_DIRS = {"current", "versions"}` in `server/reindex.py` silently excludes those two subtrees entirely (never walked, never skipped); (c) files that ARE walked but malformed land in `skipped: [{rel_path, reason}]` (reasons `filename not <YYYY-MM-DD>-<slug>.md` / `missing/invalid frontmatter: â€¦` / `bad date`). Real-repo reindex confirms `indexed: 1, removed: 0, skipped: 0` â€” clean. Removal is keyed on **file presence on disk** (a DB row whose rel_path no longer exists under a non-reserved subdir is removed), so deleting a doc â†’ `removed: 1`.

**Separately** â€” whether these workspace-internal files appear in the *published GitHub Pages site nav* is a **P3-scope** question (P3 owns Track 1 / publishing), not P2's. Filed as deferred **D1** (`Decide whether works/docs internals appear on the public site`, trigger: P3 planning, source P2.DECOMP). Do not expand P2 to solve it.
